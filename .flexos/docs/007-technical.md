# Technical Architecture — Grove

## Stack

| Layer | Choice | Rationale |
|-------|--------|-----------|
| Framework | **Nuxt 4** | SSR for marketing pages (SEO), SPA for authenticated dashboard. Vue ecosystem. File-based routing. |
| Styling | **Open Props + custom tokens** | Design token system from Doc 6 maps directly to CSS custom properties. No utility-class sprawl. |
| Database | **Convex** | Real-time reactive queries (essential for messaging inbox). Built-in auth helpers. Serverless functions for webhooks. |
| Auth | **Convex Auth** (email/password) | Simple, integrated with the database layer. No third-party auth provider needed for v1. |
| SMS | **Twilio** | Industry standard for programmable SMS. Two-way messaging, number provisioning, delivery webhooks. MMS support. |
| AI | **Claude API** (Anthropic) | Natural language response generation. System prompt per property using brand voice config + knowledge base. |
| Email | **Resend** | Transactional emails (invites, password reset, alerts). Simple API, good deliverability, reasonable free tier. |
| File Storage | **Convex File Storage** | Integrated with the database. No separate S3 config needed. Sufficient for logos and message attachments. |
| Hosting | **Vercel** | Nuxt 3 first-class support. Edge functions for Twilio webhooks. Preview deployments for development. |
| Payments | **Stripe** | Checkout for subscriptions. Customer Portal for self-serve plan changes. Webhooks for payment events. |
| Monitoring | **Sentry** | Error tracking. Performance monitoring. Reasonable free tier for early stage. |
| Analytics (internal) | **Plausible** | Privacy-friendly analytics for marketing pages. No cookie banner needed. |

## Architecture

### Rendering Strategy

| Route Type | Strategy | Why |
|-----------|----------|-----|
| Marketing pages (`/`, `/pricing`, `/about`, `/contact`) | SSR | SEO, fast first paint, social sharing meta tags |
| Auth pages (`/login`, `/signup`, `/forgot-password`, `/reset-password`) | SSR | SEO for login page, fast load |
| App pages (`/inbox`, `/guests`, `/knowledge`, `/analytics`, `/settings/*`) | SPA | Real-time data, no full page reloads, persistent WebSocket connection |
| Group pages (`/group/*`) | SPA | Same as app pages |

### Data Flow

```
Guest SMS → Twilio → Twilio Webhook
  → Convex HTTP Action (receives webhook, validates signature)
  → Convex Mutation (creates/updates records)
  → Convex Reactive Query (pushes to connected dashboard clients)
  → Dashboard updates in real time

Staff Response → Dashboard UI
  → Convex Mutation (creates message record)
  → Convex Action (calls Twilio API to send SMS)
  → Twilio → Guest phone

AI Processing → Convex Action
  → Reads: property config, knowledge base, guest history, brand voice
  → Calls Claude API with system prompt + context
  → Returns: response text + confidence score + category
  → If auto-respond: Convex Mutation → Twilio send
  → If human-needed: store draft, notify staff
```

**The browser never calls the database directly.** All mutations and queries go through Convex's server-side functions, which enforce auth, validate inputs, and manage side effects.

## Auth

### Strategy

Email/password authentication via Convex Auth.

- **Session:** JWT stored in httpOnly cookie (set by Convex). 30-day expiry. Refresh on each request.
- **Password hashing:** bcrypt with salt (handled by Convex Auth).
- **Password requirements:** Minimum 8 characters. No complexity requirements (NIST guidelines — length over complexity).
- **Reset tokens:** Cryptographically random, 1-hour expiry, single-use. Stored hashed in database.
- **Brute force protection:** 5 failed login attempts → 15-minute lockout on that email. Logged for monitoring.

### Route Protection

- Marketing and auth pages: public.
- App pages: require authenticated session. Redirect to `/login` if no session.
- Admin pages (`/analytics`, `/settings/*`): require authenticated session + admin role on current property. Redirect to `/inbox` if staff-only role.
- Group pages: require authenticated session + membership in a group-tier property.

## API Design

All API logic is via Convex functions (queries, mutations, actions). No REST API layer in v1.

### Convex Function Conventions

- **Queries** (read-only, reactive): `getConversations`, `getGuest`, `getPropertySettings`
- **Mutations** (write, transactional): `sendMessage`, `createKnowledgeEntry`, `updateGuestProfile`
- **Actions** (side effects, non-transactional): `sendSMS`, `callClaudeAPI`, `sendEmail`, `provisionPhoneNumber`

### Input Validation

All Convex functions use Convex's built-in argument validators (`v.string()`, `v.number()`, etc.). Inputs are validated before any database access.

### Error Handling

Convex functions throw `ConvexError` with structured error objects:

```typescript
throw new ConvexError({
  code: "SEAT_LIMIT_REACHED",
  message: "You've reached your plan's seat limit.",
  details: { current: 5, limit: 5, plan: "professional" }
});
```

Client catches and displays appropriate UI (toast, inline error, etc.).

## Key Integrations

### Twilio (SMS)

**What it does:** Sends and receives SMS/MMS. Provisions phone numbers.

**Inbound flow:** Twilio POSTs to `{CONVEX_URL}/twilio/inbound` on every incoming SMS. Convex HTTP Action validates the Twilio signature, extracts message body/from/media, and calls a mutation.

**Outbound flow:** Convex Action calls `twilioClient.messages.create()` with to, from (property's number), body, and optional mediaUrl.

**Number provisioning:** Convex Action calls `twilioClient.availablePhoneNumbers` to search, then `twilioClient.incomingPhoneNumbers.create()` to purchase and configure webhook URL.

**On failure:** Inbound webhook failures auto-retry (Twilio). Outbound send failures return error to the calling mutation, which marks the message as failed. Staff sees retry option.

### Claude API (AI Responses)

**What it does:** Generates natural-language guest responses and categorises requests.

**How it's called:** Convex Action constructs a system prompt including: property description, AI voice config, knowledge base entries (filtered by relevance), guest history/preferences, and current conversation context. Calls `anthropic.messages.create()`.

**System prompt structure:**
```
You are the SMS concierge for {property.name}, a boutique hotel in {property.address}.
Voice: {aiVoiceConfig.brandVoiceDescription}
Words to use: {aiVoiceConfig.wordsToUse}
Words to avoid: {aiVoiceConfig.wordsToAvoid}
Knowledge base: {relevant entries}
Guest context: {guest preferences, conversation history}
Respond as a helpful, knowledgeable hotel concierge. Keep responses under 300 characters (SMS-friendly).
Also classify this request: dining/transport/housekeeping/activities/information/complaint/other.
Also provide a confidence score 0-100.
```

**On failure:** AI service timeout (10s) or error → skip auto-response, flag conversation for human handling, log error to Sentry.

### Stripe (Payments)

**What it does:** Subscription billing, plan management, invoicing.

**Initial subscription:** Convex Action creates a Stripe Checkout Session → redirect user to Stripe-hosted checkout → on success webhook, update `properties.stripeCustomerId` and `stripeSubscriptionId`.

**Plan changes:** Redirect to Stripe Customer Portal (`stripe.billingPortal.sessions.create()`).

**Webhooks:** `customer.subscription.updated` → update plan tier. `customer.subscription.deleted` → downgrade to limited free state. `invoice.payment_failed` → send warning email.

**On failure:** Stripe webhook failures auto-retry. Payment failures trigger email to admins.

### Resend (Email)

**What it does:** Transactional emails — invites, password resets, alerts, digests.

**How it's called:** Convex Action calls `resend.emails.send()` with to, subject, HTML body (from templates).

**On failure:** Log error, retry once. If invite email fails, show "Email could not be sent" in UI — admin can retry from team settings.

## Security Checklist

- [ ] **Input validation:** All Convex function arguments validated with Convex's type system. No raw user input reaches the database.
- [ ] **XSS prevention:** Vue's template system auto-escapes by default. No `v-html` with user content. Message bodies rendered as plain text, not HTML.
- [ ] **CSRF:** Convex uses token-based auth in cookies with SameSite=Lax. No CSRF vulnerability surface for mutations.
- [ ] **Twilio webhook validation:** Every inbound webhook validated against Twilio's request signature using `twilio.validateRequest()`. Reject unsigned requests.
- [ ] **Stripe webhook validation:** All Stripe webhooks verified using `stripe.webhooks.constructEvent()` with the webhook secret.
- [ ] **Rate limiting:** Login endpoint: 5 attempts per email per 15 minutes. SMS inbound: handled by Twilio's rate limiting. AI calls: 10 per conversation per minute.
- [ ] **File upload validation:** Max 10MB. Accept only PNG, JPG, SVG (logos) or PNG, JPG (attachments). Validate MIME type server-side, not just extension.
- [ ] **Secret management:** All API keys in environment variables. Never in client bundle. Convex server-side functions only.
- [ ] **Password storage:** bcrypt hashing via Convex Auth. Never stored or logged in plaintext.
- [ ] **Data isolation:** All queries scoped to the authenticated user's property. No cross-property data access except group-tier cross-property guest lookup (scoped to same group).

## Testing Strategy

### Unit Tests (Vitest)

- Convex function logic: input validation, tier enforcement, state transitions
- AI prompt construction (given property config → expected system prompt)
- Utility functions: date formatting, timezone conversions, text truncation

### Integration Tests

- Twilio webhook → database: simulate inbound webhook, verify records created
- Stripe webhook → plan update: simulate subscription change, verify tier updated
- AI flow: mock Claude API, verify auto-response vs. human-flag logic

### E2E Tests (Playwright)

- Onboarding flow: signup → property setup → phone provisioning → first conversation
- Conversation flow: receive message (simulated) → staff responds → verify sent
- Team management: invite → accept → login → see inbox

### What's NOT Tested in v1

- Visual regression testing (manual review of prototypes is sufficient at this stage)
- Load testing (premature — optimise when approaching 50+ concurrent properties)
- Mobile native testing (no native app in v1)
