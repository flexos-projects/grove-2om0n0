# Flows & Business Logic — Grove

## Core Flows

### Flow 1: Guest Sends First Message

**Who:** Guest (SMS) → System → Staff

**Trigger:** Guest texts the hotel's Grove number for the first time.

```
Guest sends SMS to hotel's Grove number
  → Twilio webhook fires → Convex mutation
  → System checks: phone number known for this property?
  → NO (first time):
      → Create guest record (name: "Unknown Guest", phone: guest's number)
      → Create new conversation (status: active)
      → Create message record (direction: inbound, sender: guest)
      → Send greeting message to guest (from properties.greetingMessage)
      → Create message record for greeting (direction: outbound, sender: system)
      → AI evaluates the guest's message:
          → Simple request + high confidence:
              → AI sends auto-response via Twilio
              → Create message record (sender: ai)
              → Conversation stays active
          → Complex/uncertain request:
              → AI generates draft suggestion (stored on conversation, shown to staff)
              → Conversation marked unreadByStaff: true
              → Route to assignee if routing rule matches category
              → Notify assigned staff (or all staff if unassigned)
      → Dashboard updates in real time via Convex reactive query
```

**Edge cases:**
- Twilio webhook fails → Twilio retries 3 times with exponential backoff. Idempotency check on `twilioMessageSid` prevents duplicate message records.
- AI service unavailable → message flagged for human, no auto-response. Staff notification includes "AI unavailable" note.
- Greeting message already sent to this phone → don't re-send (checked via guest record existence).

**Data written:**
- `guests`: new record
- `conversations`: new record
- `messages`: 2-3 records (inbound, greeting, optional AI response)
- `properties.conversationCount`: incremented

### Flow 2: Guest Sends Subsequent Message

**Who:** Guest (SMS) → System → Staff

**Trigger:** Guest texts and already has an existing conversation.

```
Guest sends SMS
  → Twilio webhook → Convex mutation
  → System checks: phone number known? → YES
  → Find active/waiting conversation for this guest
      → EXISTS:
          → Append message to existing conversation
          → Update conversation: lastMessageAt, lastMessagePreview, status → active
          → AI evaluates → auto-respond or flag for human (same as Flow 1)
      → NO active conversation (previous was resolved):
          → Create new conversation
          → Same flow as first message (minus greeting)
  → Guest profile: update lastContactAt
  → Notify assigned staff member (or all if unassigned)
```

**Edge cases:**
- Guest sends multiple messages rapidly → each creates a message record, AI evaluates the latest one as a batch (waits 5 seconds for additional messages before processing).
- Conversation was resolved → new conversation created, not reopened.

### Flow 3: Staff Responds to Guest

**Who:** Staff → System → Guest (SMS)

**Trigger:** Staff types a response and clicks Send in the inbox.

```
Staff types message in conversation thread
  → Option A: Staff uses AI draft suggestion (one tap, editable)
  → Option B: Staff writes custom response
  → Option C: Staff selects quick action (editable before send)
  → Staff clicks Send
      → Create message record (direction: outbound, sender: staff, senderUserId: staff's ID)
      → Send SMS via Twilio API
      → Update message: twilioMessageSid, deliveryStatus: queued
      → Update conversation: lastMessageAt, lastMessagePreview, status → waiting
      → Mark conversation: unreadByStaff: false
  → Twilio delivery webhook:
      → Update message: deliveryStatus → sent → delivered (or failed)
```

**Edge cases:**
- SMS send fails (Twilio error) → message marked `deliveryStatus: failed`, staff sees inline error with retry button.
- Staff sends while guest is simultaneously texting → both messages appear in chronological order, no conflict (Convex handles concurrent mutations).
- Image attachment → uploaded to Convex storage, URL included in Twilio MMS.

**Data written:**
- `messages`: new record
- `conversations`: updated (lastMessageAt, lastMessagePreview, status, unreadByStaff)

### Flow 4: Staff Adds Internal Note

**Who:** Staff

**Trigger:** Staff toggles "Internal Note" mode and writes a note.

```
Staff clicks internal note toggle
  → Input field changes to yellow/amber styling
  → Staff types note and submits
  → Create message record (isInternalNote: true, direction: outbound, sender: staff)
  → Note appears in conversation thread with "Staff Only" label
  → NO SMS is sent — this never touches Twilio
  → Input returns to normal mode
```

**Data written:**
- `messages`: new record (isInternalNote: true)

### Flow 5: Hotel Onboarding

**Who:** Hotel operator (new user)

**Trigger:** Operator clicks "Start Free Trial" on landing page.

```
Step 1: Account Creation
  → Email, password, full name
  → Validate email uniqueness
  → Create user record
  → Create session
  → → Step 2

Step 2: Property Setup
  → Property name, address (Google Places autocomplete), timezone (auto-detect from address)
  → Description (optional, used for AI context)
  → Create property record (planTier: starter, conversationLimit: 500)
  → Create propertyMember record (role: admin)
  → → Step 3

Step 3: Phone Number Provisioning
  → User selects preferred area code
  → System searches Twilio for available numbers in that area code
  → Display 3 options → user picks one
  → Purchase number via Twilio API
  → Configure Twilio webhook to point to Grove's inbound endpoint
  → Update property: phoneNumber, twilioSid
  → → Step 4

Step 4: Test Drive
  → Display the provisioned number prominently
  → "Text this number to see Grove in action"
  → User sends a test SMS → AI responds with a demo greeting
  → UI shows the conversation appearing in real time
  → "Continue to Dashboard" button
  → → Redirect to /inbox with welcome overlay

  → Seed default quick actions for the property
  → Seed 3 sample knowledge entries (customisable)
```

**Edge cases:**
- Twilio number purchase fails → show error, offer retry with different area code, or fallback to manual provisioning (support contact).
- User abandons mid-onboarding → partial records exist. On next login, detect incomplete onboarding and resume from the last incomplete step.
- Email already exists → show login link with message "This email already has an account."

**Data written:**
- `users`: new record
- `properties`: new record
- `propertyMembers`: new record
- `quickActions`: 5 default records
- `knowledgeEntries`: 3 sample records

### Flow 6: Team Invitation

**Who:** Admin → Invitee

**Trigger:** Admin enters email and role in team settings.

```
Admin enters email + selects role (admin/staff)
  → Check seat limit for property's plan tier
      → At limit: show error "You've reached your seat limit. Upgrade to add more team members."
      → Under limit:
          → Create invite record (status: pending, expiresAt: +7 days)
          → Send invitation email with unique link
          → Invite appears in team list as "Pending"
  → Invitee clicks email link:
      → If invitee has account: add propertyMember record, mark invite accepted, redirect to /inbox
      → If invitee is new: show signup form (name, password), create user, add propertyMember, mark invite accepted, redirect to /inbox
  → After 7 days: invite status → expired. Admin can re-send.
```

**Edge cases:**
- Invitee email already a member → show error "This person is already on your team."
- Invite link used after expiry → show message with "Request a new invite from your admin."

### Flow 7: Conversation Assignment & Routing

**Who:** System (auto) or Staff (manual)

**Trigger:** New message arrives or staff manually assigns.

```
Auto-routing (on new message):
  → AI categorises the message (dining, transport, etc.)
  → Check routingRules for this property + category
      → Rule exists: assign conversation to specified user
      → No rule: leave unassigned (general queue)
  → Assigned user gets notification

Manual assignment:
  → Staff clicks assignee avatar on conversation
  → Dropdown shows all team members
  → Staff selects → conversation.assignedToUserId updated
  → New assignee gets notification
  → Previous assignee (if any) sees "reassigned" note
```

### Flow 8: Password Reset

**Who:** User

**Trigger:** User clicks "Forgot Password" on login page.

```
User enters email → clicks "Send Reset Link"
  → If email exists: generate reset token (expires 1 hour), send email
  → If email doesn't exist: show same "Check your email" message (no leak)
  → User clicks link in email
  → If token valid: show new password form → update password hash → redirect to login
  → If token expired: show message "This link has expired" with link to request new one
```

## State Machines

### Conversation Status

```
[new message from guest]
      ↓
   active ←→ waiting
      ↓         ↓
   resolved  resolved
      ↓
   (new message from guest → new conversation)
```

| Transition | Trigger |
|-----------|---------|
| → active | New inbound guest message |
| active → waiting | Staff sends response |
| waiting → active | Guest sends new message |
| active → resolved | Staff clicks Resolve |
| waiting → resolved | Staff clicks Resolve |
| resolved → (new conversation) | Guest sends message after resolution |

### Invite Status

```
pending → accepted
pending → expired (after 7 days)
```

### Message Delivery Status

```
queued → sent → delivered
queued → failed
sent → failed
```

## Error States

| Flow | Error Condition | Response |
|------|----------------|----------|
| Inbound SMS | Twilio webhook timeout | Twilio auto-retries. Idempotency on message SID. |
| Inbound SMS | AI service down | Flag for human, notify staff "AI unavailable." |
| Outbound SMS | Twilio send failure | Mark message failed, show retry button to staff. |
| Outbound SMS | Guest number blocked/invalid | Mark message failed, show "Unable to deliver" with reason. |
| Onboarding | Number provisioning fails | Retry with different area code, or support contact. |
| Onboarding | Incomplete signup | Resume from last step on next login. |
| Login | Wrong password 5x | Lock account 15 minutes, show message. |
| Team invite | Seat limit reached | Block invite with upgrade CTA. |
| Knowledge base | Entry limit reached (Starter) | Block add with upgrade CTA. |
| Conversation limit | Monthly limit reached (Starter) | AI stops auto-responding, staff can still manually respond. Banner in dashboard. |
